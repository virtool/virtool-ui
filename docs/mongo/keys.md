# keys

User-issued API keys. Each document is one key — `_id` is the SHA-256 hash of the raw secret, `id` is a human-readable alternate id (`<lowercase-name>_<n>`) used in the API. Authentication looks the document up by hashed secret, intersects its `permissions` with the owner's current permissions at request time, and treats matched keys as that user.

## Source

- Python module: `virtool/account/{data,api,models,oas}.py` — the entire keys data layer lives in `AccountData` (`get_keys`, `get_key`, `get_key_by_secret`, `create_key`, `update_key`, `delete_key`, `delete_keys`).
- Mongo binding: `virtool.mongo.core.Mongo.keys` → collection name `keys`.
- Pydantic model (response shape): `virtool/account/models.py::APIKey` (`id`, `created_at`, `groups: list[GroupMinimal]`, `name`, `permissions: Permissions`).
- Live schema: production `$jsonSchema` validator (Compass export, supplied during port).

## Document shape

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `_id` | `string` | yes | SHA-256 hex of the raw secret (`virtool.utils.hash_key`). The raw secret is returned to the client exactly once at create time and never stored. Lookups by secret hash this server-side. **Not** an 8-char alphanumeric id like other collections. |
| `id` | `string` | yes | Alternate id used everywhere in the API (request paths, response bodies). Generated by `_get_alternate_id` as `f"{name.lower()}_{suffix}"` where `suffix` increments until the candidate is not in `mongo.keys.distinct("id")`. Globally unique across the collection (not scoped to a user). |
| `name` | `string` | yes | Human-readable label set by the user at create time. No uniqueness constraint. |
| `created_at` | `Date` | yes | UTC, set from `virtool.utils.timestamp()` on insert. |
| `permissions` | `Permissions` | yes | Object with the eight boolean permission flags (`cancel_job`, `create_ref`, `create_sample`, `modify_hmm`, `modify_subtraction`, `remove_file`, `remove_job`, `upload_file`). All eight are required by the live validator. Defaults from `Permissions()` are all `false`. The Python `Permissions` Pydantic model has all-`False` defaults but the writer uses `data.permissions.dict(exclude_unset=True)` — keys created via the API may insert with only the fields the request set; the live validator's `required` list says prod docs always carry all eight, suggesting Pydantic defaults are filled before insert (the request goes through `CreateKeyRequest.permissions: Permissions` which materialises the full object). Modelled as a sub-schema with all eight booleans `required`. |
| `groups` | `(int \| string)[]` | yes | Snapshot of the owner's group ids at create time (`[group.id for group in user.groups]`). The Pydantic `GroupMinimal.id` is typed `int \| str`, so legacy string handles can appear. The live validator confirms `items: { bsonType: ["int", "string"] }`. Read paths feed this through `AttachGroupsTransform` to expand into `GroupMinimal[]` for the API response. The stored value is *not* updated when the user's group membership changes — it's a snapshot, but the read path attaches current group records by id. |
| `user` | `{ id: int \| string }` | yes | Owner. Live validator allows both `int` and `string` (legacy MongoDB user handle). Reads use `get_user_id_single_variants` to query both forms during the migration. Modelled as `Mixed` to round-trip pre-backfill string ids; tighten to `Number` once the user-id backfill is complete. |
| `administrator` | `boolean` | no | Legacy field. Present on some prod docs (in `properties` but **not** `required` per the live validator). The current `create_key` writer does not set it, and no read path consults it — administrator status is resolved at auth time from the live user's `administrator_role`, not from the key. Kept on the schema as optional so it round-trips; safe to leave unset on new writes. |

## Authoritative side

Mongo-only. No Postgres counterpart. The owner's group memberships and live permissions are joined in from Postgres at read time (`AttachGroupsTransform`, plus `limit_permissions(key.permissions, user.permissions)` in `get_key`).

## Quirks captured from Python

- **`_id` is the secret.** Authentication (`get_key_by_secret`) hashes the incoming raw key with `hash_key` and looks it up by `_id`. Default projections strip `_id` (and `user`) before returning the document — the API response uses `id` as the public identifier. Never include `_id` in API responses.
- **`id` is globally unique, not user-scoped.** `_get_alternate_id` runs `mongo.keys.distinct("id")` across the *whole collection* and increments until it finds a free `<name>_<n>`. Two users creating keys named "ci" produce `ci_0` and `ci_1`. Don't introduce a per-user uniqueness assumption.
- **`update_key` reads-then-writes permissions.** The data layer pulls the existing `permissions` map, merges `data.permissions.dict(exclude_unset=True)` over it, and `$set`s the result. So a partial update preserves bits the request didn't mention. Replicating this naively with a top-level `$set: { permissions: {...} }` is fine, but nested `$set` of individual permission flags is also valid.
- **Reads intersect with live user permissions.** `get_key` (non-admin path) calls `limit_permissions(document["permissions"], user.permissions.dict())` so the response can never show a permission the user no longer has. The stored value is the original snapshot; the served value is intersection. New API endpoints that surface key data must apply the same intersection or the response leaks revoked permissions.
- **Groups are stored as a snapshot but rendered live.** The stored `groups: [id, ...]` is the user's group ids at the moment of creation, but `AttachGroupsTransform` re-fetches each group from Postgres at read time. If a group is renamed or deleted, the response reflects the current Postgres state — not the snapshot. If a group id is stale/missing in Postgres, behaviour is whatever the transform does on a miss (most likely drops it).
- **No application-managed indexes.** `virtool/mongo/core.py` does not call `create_index` on `keys`. Only the implicit `_id` index exists. The hot query is `find_one({_id: hashed})` (auth path) which uses it directly. Other reads are `find({user.id: {$in: variants}})` (per-user listing) and `find_one({id, user.id: ...})` (single key by alternate id). At current scale (one collection-wide read per auth, low key counts per user) this hasn't been a problem; if it grows, a `{ "user.id": 1 }` index would be the obvious add. Don't add one speculatively.
- **`delete_keys` is bulk-by-user.** Used during password change / session invalidation flows (`virtool/account/data.py`) to wipe every key the user owns. Modelled at the data-layer level, not at the schema level — but worth knowing when porting account flows.
- **`administrator` is dead from the writer side.** Unset on every key created by the current code path. The handful of prod docs that carry it likely predate the Pydantic-driven create path. Treat as read-only legacy.

## Mongoose schema

`apps/web/src/server/db/mongo/keys.ts`. `strict: false` so legacy fields (notably `administrator`) round-trip, explicit `collection: 'keys'`, no auto timestamps.

Inferred type: `KeyDoc` (`InferSchemaType<typeof keySchema>`).

Mixed-typed fields (`Schema.Types.Mixed`):

- `user.id` — `int | string` until the user-id backfill is complete.
- `groups` items — `int | string` for the same reason (legacy group handles).

Sub-schema `keyPermissionsSchema` is exported for reuse if other documents grow a permissions snapshot. All eight bits are `required`; Pydantic defaults fill them on the way in, and the live validator's `required` list confirms prod docs carry the full set.

## Open follow-ups

- Drop `user.id` and `groups[]` from `Mixed` to `Number` / `Number[]` once the user-id and group-id backfills land. `samples` and `indexes` will need the same loosening lifted at the same time — keep them in sync.
- Decide whether `administrator` should be retired (rewritten out of the prod docs that have it) or kept as a no-op round-trip field. No code reads it today, so it's a candidate for the dropped-fields list after a prod scan confirms how rare it is.
- Consider a `{ "user.id": 1 }` index if per-user listing becomes a hot path. Not needed today.
- The two-token (`_id` + `id`) shape works but is a footgun — every read needs a `{ _id: false }` projection. Worth a comment at the data-layer call sites in the new app.
